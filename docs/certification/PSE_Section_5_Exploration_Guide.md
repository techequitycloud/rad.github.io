# PSE Certification Preparation Guide: Exploring Section 5 (Supporting compliance requirements)

This guide is designed to help candidates preparing for the Google Cloud Professional Cloud Security Engineer (PSE) certification. It focuses specifically on Section 5 of the exam guide (which covers ~11% of the exam). By using your web-based deployment portal to configure Terraform variables for the available modules and then exploring the Google Cloud Platform (GCP) console, you will gain hands-on context for these critical architectural topics related to compliance.

---

## 5.1 Adhering to regulatory and industry standards requirements for the cloud

### Concept
Evaluating the shared responsibility model, mapping compliance requirements to controls, and determining technical needs relative to compute, data, network, and storage.

**Evaluating the shared responsibility model:**
By using managed services like Cloud Run and GKE via your Terraform modules, you shift significant operational and security responsibilities to Google, simplifying compliance efforts.
*   **Exploration**: Google manages the underlying hardware, hypervisors, and control plane (in the case of GKE Autopilot), allowing you to focus on application security, IAM policies, and data protection through your module configurations.
*   **Customization Suggestions**: Review the documentation for GKE Autopilot vs. GKE Standard. If your deployment portal exposes cluster type configurations, try changing a cluster to Autopilot to observe the shift in operational responsibilities and default security postures.

### Exploration
*   Review the architecture in your project. Navigate to the **Security Command Center** (if enabled in your organization) to view compliance posture dashboards and see how the deployed resources score against standard frameworks like NIST 800-53 or CIS Foundations Benchmark.

#### 1. VPC Isolation & Network Segmentation (Restricting compute and data)
*   **Concept**: Segmenting network traffic to limit the blast radius of a potential breach and restricting access strictly to authorized resources is a core requirement of many compliance standards.
*   **Implementation**: Network restrictions can be enabled via your module variables. For GKE modules, enabling micro-segmentation configures Kubernetes Network Policies. For serverless (Cloud Run) modules, variables controlling VPC connectors and ingress settings (e.g., `ingress = "INGRESS_TRAFFIC_INTERNAL_AND_CLOUD_LOAD_BALANCING"`) ensure traffic does not traverse the public internet unnecessarily.
*   **Exploration**:
    - In the GCP Console, navigate to **Kubernetes Engine > Network Policies** to visualize active policies.
    - Go to **VPC network > VPC network peering** to explore how VPCs might connect, and **Serverless VPC Access** to review connector settings deployed by your modules.
*   **Customization Suggestions**: Through your deployment portal, adjust the network segmentation variables for a GKE module to observe how external ingress is blocked. For a Cloud Run module, try toggling the VPC connector variables to see how outbound traffic behaves.

#### 2. Secret Management (Data protection)
*   **Concept**: Storing sensitive information like database credentials and API keys securely and rotating them to meet compliance requirements.
*   **Implementation**: Your Terraform modules integrate with Google Secret Manager. Secrets are never hardcoded; instead, they are provided as sensitive variables through your deployment portal, stored in Secret Manager, and retrieved dynamically by the applications.
*   **Exploration**:
    - In the GCP Console, navigate to **Security > Secret Manager**. Click on a secret generated by your deployment to view its versions and applied IAM policies.
    - Check the **Permissions** tab on a specific secret to see which service accounts have `roles/secretmanager.secretAccessor`.
*   **Customization Suggestions**: Add a new sensitive variable via your deployment portal and observe how it creates a new entry in Secret Manager. Explore any module variables related to automated credential rotation schedules (like Cloud Run Jobs or Kubernetes CronJobs) if available.

#### 3. Identity-Aware Proxy (IAP) (Access Control)
*   **Concept**: Enforcing zero-trust access by requiring identity verification (beyond just network presence) before granting access to applications.
*   **Implementation**: You can enable IAP through your module's `enable_iap` variable (often found in security or access control groups in your portal). This restricts access to internal tools by verifying user identity and context.
*   **Exploration**:
    - In the GCP Console, go to **Security > Identity-Aware Proxy**. Observe the resources protected by IAP.
    - Select a resource and view the information panel on the right. Notice how the `IAP-secured Web App User` role is granted to specific identities or groups you configured in the portal.
*   **Customization Suggestions**: In your deployment portal, modify the `trusted_users` variable (or similar allowed list) by adding a new Google Group and applying the changes. Explore Access Context Manager in the GCP Console to create context-aware access levels and attach them to the IAP configuration.

#### 4. Cloud Armor (Web Application Firewall - WAF)
*   **Concept**: Protecting applications against common web exploits (like OWASP Top 10) and distributed denial-of-service (DDoS) attacks.
*   **Implementation**: Your Terraform modules can integrate with Cloud Armor (typically via an `enable_cloud_armor` variable). When enabled, Cloud Armor security policies enforce rate limiting, block malicious IPs, and mitigate common web vulnerabilities.
*   **Exploration**:
    - In the GCP Console, navigate to **Network Security > Cloud Armor**.
    - Click on the policy applied by your deployment to inspect the individual rules (e.g., SQLi, XSS protections) and review the priority of each rule.
*   **Customization Suggestions**: If your portal exposes Cloud Armor rule configurations (like IP blocklists), add a custom IP range and apply the configuration. Enable Adaptive Protection (if supported by your billing tier) and monitor the logs for potential layer 7 attacks.

#### 5. Binary Authorization (Supply Chain Security)
*   **Concept**: Ensuring that only trusted, verified, and signed container images are deployed to the environment.
*   **Implementation**: Your modules support Binary Authorization, which ensures that images are signed by a trusted authority (e.g., via Cloud KMS) before they are instantiated, preventing unverified code from running in your clusters or serverless environments.
*   **Exploration**:
    - In the GCP Console, go to **Security > Binary Authorization**.
    - Review the policy enforced by your module. See how attestors are configured and how they map to your specific projects or namespaces.
*   **Customization Suggestions**: If your deployment portal exposes Binary Authorization toggles, try enabling it for a new namespace and attempt to deploy an unsigned image (bypassing the standard CI/CD pipeline) to see the rejection in action within the GCP Console.

---

By configuring Terraform variables in your deployment portal and exploring the resulting architecture in the Google Cloud console, you can see how GCP features are practically applied to enforce security controls and meet compliance requirements, directly aligning with the expectations of the Professional Cloud Security Engineer exam.
