# PCA Certification Preparation Guide: Section 3 — Designing for security and compliance (~17.5% of the exam)

This guide helps candidates preparing for the Google Cloud Professional Cloud Architect (PCA) certification explore Section 3 of the exam through the lens of the Tech Equity RAD platform at [https://radmodules.dev](https://radmodules.dev). Three modules are relevant to this section: **GCP Services**, which establishes the foundational shared infrastructure; **App CloudRun**, which deploys serverless containerised applications on Cloud Run; and **App GKE**, which deploys containerised workloads on GKE Autopilot.

You interact with each module by configuring its variables in the RAD UI deployment portal, then exploring the resulting infrastructure in the GCP Console. This guide maps each exam topic to the relevant variables you can configure and the console locations where you can observe the outcomes. It also highlights PCA objectives that are *not* currently implemented by these modules, providing guidelines for self-guided research and exploration.

---

## 3.1 Designing for security

### Identity and Access Management (IAM) and Separation of Duties
**Concept:** Applying the principle of least privilege using predefined IAM roles.

**In the RAD UI:**
*   **Least Privilege:** The platform natively implements separation of duties by provisioning dedicated custom Service Accounts (e.g., `cloud_run_sa` vs `gke_sa` vs `cloud_build_sa`) rather than using the default compute service account.
*   **Access Control:** The `support_users` variable (Group 1) allows mapping Workspace Groups to specific viewer/monitoring roles, automating lifecycle management through Cloud Identity.

**Console Exploration:**
Navigate to **IAM & Admin > IAM** to view the assigned roles. Navigate to **IAM & Admin > Service Accounts** and view the "Permissions" tab.

**Real-world example:** A professional services firm enforces separation of duties by provisioning a dedicated Cloud Run service account with only `roles/cloudsql.client` and `roles/secretmanager.secretAccessor` — the minimum permissions required to query the database and retrieve credentials. The default Compute Engine service account, which carries broad `Editor` permissions, is explicitly disabled on all new projects via an Organization Policy to prevent accidental over-permissioning.

### Data Security and Secure Remote Access
**Concept:** Protecting sensitive data from unauthorized access or exfiltration using encryption and zero-trust proxies.

**In the RAD UI:**
*   **Secret Management:** The `enable_auto_password_rotation` variable (Group 11 for Cloud Run; §7.D Auto Password Rotation in App GKE) automates credential rotation, securely passing them to workloads via Secret Manager. The `secret_rotation_period` variable controls how frequently Secret Manager emits a rotation notification event (default: 30 days).
*   **Secrets Store CSI Driver (GKE):** Enable `enable_secrets_store_csi_driver` (§4.E Advanced Security in App GKE) to fetch secrets directly from Secret Manager at pod start time without storing plaintext values in Kubernetes Secrets or Terraform state — the most secure secret delivery method for GKE workloads with strict compliance requirements such as PCI-DSS or HIPAA. The module activates the GKE Secrets Store CSI Driver on the cluster and creates a `SecretProviderClass` manifest per namespace mapping each entry in `secret_environment_variables` to a Secret Manager secret version.
*   **Identity-Aware Proxy (IAP):** Review the `enable_iap` variable (Group 15 for Cloud Run; §4.B Identity-Aware Proxy in App GKE). IAP provides secure remote access, replacing traditional VPNs by verifying Google identities at the edge. In App GKE, IAP requires a pre-created OAuth 2.0 client (`iap_oauth_client_id` / `iap_oauth_client_secret`) because it is enforced via a `GCPBackendPolicy` on the Gateway backend.
*   **Network Segmentation (GKE):** The `enable_network_segmentation` variable (§3.D Networking & Network Policies in App GKE) creates Kubernetes NetworkPolicy objects using GKE Dataplane V2 (Cilium), restricting east-west traffic to only explicitly permitted connections. Egress to Private Google Access endpoints (`restricted.googleapis.com`, `private.googleapis.com`) is explicitly permitted, making this configuration compatible with VPC Service Controls environments.
*   **VPC Service Controls:** In **GCP Services**, `enable_vpc_sc` (Group 10) creates a VPC Service Perimeter, preventing data exfiltration by blocking API access outside the defined trusted boundary.
*   **Securing Software Supply Chain:** In **GCP Services**, `enable_binary_authorization` (Group 11) ensures only verified, signed container images are deployed.

**Console Exploration:**
Navigate to **Security > Secret Manager** to view secret versions. Navigate to **Security > VPC Service Controls** to see the perimeter configuration. Navigate to **Security > Binary Authorization**.

**Real-world example:** A healthcare organization handles patient data subject to HIPAA requirements. They define a VPC Service Perimeter around the Cloud Healthcare API project so that even a compromised service account cannot exfiltrate data to an external Cloud Storage bucket outside the perimeter. Binary Authorization is enforced so that only container images that have passed a Cloud Build vulnerability scan and been attested by the security team can be deployed to production GKE clusters — preventing a supply-chain attack where a malicious dependency is introduced in a CI build.

### 💡 Additional Security Objectives & Learning Guidelines
*   **App_Common Three-Layer Architecture:** The platform is structured as three composable layers: the App_Common module (shared sub-modules: `app_secrets`, `app_monitoring`, `app_cicd_base`, `app_cloud_deploy`, `app_cmek`, `app_security`, `app_storage_enhanced`, `app_db_clients`, `app_dashboard`, and others); the Platform layer (App_CloudRun or App_GKE, which composes App_Common sub-modules); and the Application wrapper (a thin Terraform module per environment that passes variables to the Platform layer). This architecture is the key design pattern for understanding the RAD platform: the Platform layer is never deployed directly — it is always invoked from an Application wrapper that supplies environment-specific variables. This separation of concerns allows multiple application teams to share the same Platform layer implementation while independently managing their own variable configurations.
*   **Cloud Run Ingress Settings and Cloud Armor Pairing:** The `ingress_settings` variable (Group 14 in App CloudRun) must be set to `internal-and-cloud-load-balancing` when `enable_cloud_armor = true` — this is an architectural requirement, not merely a best practice. Without this pairing, the Cloud Run service URL is directly reachable from the internet, completely bypassing the Cloud Armor WAF. In security architecture terms, this is a "defence in depth gap": Cloud Armor provides the WAF layer, but `ingress_settings` enforces that the WAF is the only path to the application. Architects must specify this pairing explicitly in any design that includes both Cloud Armor and Cloud Run.
*   **GCS Fuse CSI Driver for Stateful Workloads:** App GKE supports the GCS Fuse CSI Driver via the `gcs_volumes` variable (§3.C Storage). The driver mounts Cloud Storage buckets directly as POSIX-compatible filesystems inside GKE pods — applications read and write files using standard file I/O without any object storage client library. The cluster annotation `gke-gcsfuse/volumes: "true"` on the pod spec activates the sidecar that handles GCS bucket mounting. This is architecturally significant: it enables stateful workloads (ML model serving, media processing, document rendering) to access Cloud Storage data as if it were local disk, without modifying application code to use the GCS client libraries.
*   **Cloud Run Gen2 Execution Environment:** The `execution_environment` variable (Group 3 in App CloudRun) selects between the `gen1` and `gen2` Cloud Run execution environments. Gen2 is required for workloads using NFS mounts, GCS Fuse mounts (via `FUSE`), Unix sockets, or multi-threaded processing that needs full Linux compatibility. From an architecture perspective, the choice of execution environment determines which storage backends a Cloud Run service can access — gen1 is suitable for stateless HTTP handlers, while gen2 is required for any workload that needs to mount shared storage or interact with the Linux kernel at a deeper level.
*   **gRPC and HTTP/2 Protocol Support:** The `container_protocol` variable (Group 3 in App CloudRun) can be set to `"h2c"` to enable end-to-end HTTP/2 cleartext (gRPC) for the Cloud Run service. This is required for backend gRPC services — without `h2c`, the load balancer downgrade to HTTP/1.1 breaks gRPC streaming calls. Architects designing microservice communication patterns must specify `container_protocol = "h2c"` for any Cloud Run service that exposes a gRPC API, and configure the client to use gRPC transport with h2c negotiation.
*   **Customer-Managed Encryption Keys (CMEK):** Practice creating a Cloud KMS Key Ring and Key. Then, encrypt a Cloud Storage bucket or a Compute Engine persistent disk using that specific CMEK. Understand that CMEK gives you control over key rotation and revocation, and that revoking a CMEK immediately renders the encrypted data inaccessible — a critical capability for data destruction under GDPR right-to-erasure requirements.

    **In the RAD UI:** App CloudRun (Group 9 `manage_storage_kms_iam = true`) and App GKE (via the shared `App_Common/modules/app_cmek` sub-module, composed by `app_storage_wrapper`) both support CMEK-encrypted Cloud Storage buckets. The `app_cmek` module provisions a KMS keyring named `${project_id}-cmek-keyring` and a `storage-key` CryptoKey. Set `manage_storage_kms_iam = true` in App CloudRun Group 9 on a second deployment (after the KMS key exists in the project) to activate bucket-level CMEK encryption. Navigate to **Security > Key Management** to observe the keyring and to **Cloud Storage > Buckets > Configuration** to confirm `Customer-managed key` is shown as the encryption type. Binary Authorization in App GKE (§4.C) uses a *separate* asymmetric CMEK keyring (`${project_id}-binauthz-keyring`) for image attestation signing — navigate to **Security > Key Management** to see both keyrings after a full deployment.
    
    The `app_storage_enhanced` sub-module (App_Common) applies advanced security controls to every Cloud Storage bucket: `public_access_prevention: enforced` blocks all public access ACLs regardless of bucket-level permissions, and soft-delete policies retain deleted objects for a configurable retention window — protecting against accidental or malicious data deletion while keeping the data irrecoverable to external principals.
*   **Resource Hierarchy and Organization Policy:** Understand how Organization Policies enforce constraints that flow down from Organization to Folders to Projects. Key constraints to know include `compute.requireShieldedVm`, `iam.disableServiceAccountKeyCreation`, `compute.restrictCloudRunRegions`, and `gcp.resourceLocations` for data residency. Projects inherit policies from parent folders automatically and cannot override a denied policy set at a higher level.
*   **Hierarchical Firewall Policy:** Unlike VPC firewall rules (which are per-network), Hierarchical Firewall Policies are attached at the Organization or Folder level and evaluated before VPC firewall rules. This allows a central security team to enforce baseline deny rules (e.g., block all RDP/SSH from the internet) across all projects in a folder, without relying on individual project teams to configure VPC rules correctly.
*   **Context Aware Access and Chrome Enterprise Premium:** Beyond IAP, study how Chrome Enterprise Premium (formerly BeyondCorp Enterprise) enables context-aware access policies based on device trust level, geographic location, and risk score — providing zero-trust access to web applications without a VPN. For example, a policy can require that only managed corporate devices with an up-to-date OS can access sensitive internal applications, while personal devices are denied even if the user's identity is valid.
*   **Service Account Impersonation:** Understand how to use service account impersonation (`iam.serviceAccounts.actAs`) to allow a user or workload to temporarily act as a target service account and call APIs with its permissions — without downloading a long-lived JSON key. This pattern is preferred for developer workflows (e.g., running Terraform locally as a service account) because the impersonation token expires automatically.
*   **Workload Identity Federation:** Research how to authenticate GitHub Actions CI pipelines or workloads running on AWS EC2 directly to GCP without downloading long-lived JSON service account keys, using short-lived federated tokens instead.
*   **Additional Services as Independently Managed Deployments:** App GKE's `additional_services` variable (§3.F Core Services) deploys helper microservices as independent Kubernetes Deployments alongside the primary application — each with its own `container_image`, resource requests, HPA, and the option to export its ClusterIP via `output_env_var_name`. For architects, this is the recommended pattern for sidecar-style services (background workers, cache warming services, internal APIs) that should scale independently from the primary application. App CloudRun's `additional_services` (Group 6) provides the equivalent for serverless: each additional service is a separate Cloud Run service with `INGRESS_TRAFFIC_INTERNAL_ONLY`, reachable only from within the VPC — not from the public internet.
*   **Cloud Deploy Cross-Project and Cross-Region Stages:** The `cloud_deploy_stages` variable (Group 7 for App CloudRun; §6.B for App GKE), via the App_Common `app_cloud_deploy` sub-module, supports per-stage `project_id` and `region` overrides. This enables promotion pipelines that span GCP project and region boundaries — for example, deploying to a developer's sandbox project in `us-central1` for the `dev` stage, a shared staging project in `us-east1` for `staging`, and a hardened production project in `europe-west1` for `prod`. Each stage target uses separate service account impersonation, ensuring that the CI/CD pipeline's service account cannot self-promote to production — the production deployment requires a dedicated, restricted service account with access only to the production project.
*   **Inline Infrastructure for Self-Contained Deployments:** App GKE's Inline Infrastructure mode (§9) provisions the complete GCP environment — VPC, GKE Autopilot cluster, Cloud Filestore NFS, and Cloud SQL — from within the module's Terraform state using `prereq_needs_*` flags. This architecture is appropriate for team-scoped or feature-branch environments where the complete infrastructure lifecycle is owned by a single deployment unit. For architects, the trade-off is between simplicity (all resources in one state file) and modularity (shared platform infrastructure managed separately from application resources).
*   **Securing AI:** Study Model Armor and Sensitive Data Protection (DLP API) to secure model deployment and sanitize training data.

## 3.2 Designing for compliance

### Audits and Industry Certifications
**Concept:** Ensuring the architecture adheres to industry standards (PCI-DSS, HIPAA, SOC 2) and logging all activities.

**In the RAD UI:**
*   **Security Command Center:** In **GCP Services**, `enable_security_command_center` (Group 16) activates SCC, continuously scanning for misconfigurations against compliance standards like SOC 2 and PCI-DSS.

**Console Exploration:**
Navigate to **Security > Security Command Center** to view the compliance dashboard and vulnerabilities.

**Real-world example:** A payment processor undergoing PCI-DSS audit uses Security Command Center's compliance posture view to generate evidence that no public-facing Cloud Storage buckets exist, no service accounts have owner-level permissions, and all Compute Engine instances use encrypted disks. SCC automatically maps its findings to the relevant PCI-DSS control IDs, reducing the manual effort required to produce audit artefacts.

### 💡 Additional Compliance Objectives & Learning Guidelines
*   **Cloud Audit Logs:** Review the difference between Admin Activity logs (enabled by default) and Data Access logs (must be manually enabled, high volume). Practice exporting Data Access logs to BigQuery for forensic analysis.
*   **Legislation and Regulation:** Understand how data residency and sovereignty requirements impact region selection. Research Assured Workloads for government/regulated deployments.
*   **Sensitive Data Protection:** Explore how the DLP API handles PII (personally identifiable information) masking in real-time or via storage scans.
