# Cal.diy on GKE Autopilot

This document provides a comprehensive reference for the `modules/Cal_GKE` Terraform module. It covers architecture, IAM, configuration variables, Cal.diy-specific behaviours, and operational patterns for deploying Cal.diy on Google Kubernetes Engine (Autopilot).

---

## 1. Module Overview

Cal.diy is the MIT-licensed, self-hostable fork of Cal.com — the scheduling platform used by millions globally to eliminate the back-and-forth of meeting coordination. `Cal_GKE` is a **wrapper module** built on top of `App_GKE`. It uses `App_GKE` for all GCP and Kubernetes infrastructure provisioning and injects Cal.diy-specific application configuration, database initialisation, and secret management via `Cal_Common`.

**Key Capabilities:**
*   **Compute**: GKE Autopilot, Next.js 14 container, 2 vCPU / 2 Gi by default. Horizontal Pod Autoscaling with `min_instance_count = 1` and `max_instance_count = 5`.
*   **Data Persistence**: Cloud SQL **PostgreSQL 15**. A `data` GCS bucket is provisioned by default.
*   **Security**: `NEXTAUTH_SECRET` and `CALENDSO_ENCRYPTION_KEY` auto-generated by `Cal_Common` and stored in Secret Manager. Workload Identity binds the GKE pod SA to a GCP SA with the minimum required permissions.
*   **Public URL Wiring**: `NEXT_PUBLIC_WEBAPP_URL` and `NEXTAUTH_URL` are set to `$(GKE_SERVICE_URL)` — a sentinel value that `App_GKE`'s `replace-placeholder.sh` resolves to the actual LoadBalancer external IP or custom domain at apply time.
*   **CI/CD**: Custom Cloud Build image pipeline by default; Cloud Deploy progressive delivery optional.
*   **Reliability**: Health probes target `/api/health`. Generous probe timeouts accommodate Prisma migrations on first boot.

**Wrapper architecture:** `Cal_GKE` calls `Cal_Common` to build an `application_config` object containing Cal.diy-specific secrets, probe configuration, and the `db-init` job definition. The `GKE_SERVICE_URL` sentinel in `NEXT_PUBLIC_WEBAPP_URL` and `NEXTAUTH_URL` is resolved by `App_GKE` at apply time — the actual LoadBalancer IP or custom domain is injected after the Service resource is created. `module_secret_env_vars` carries `NEXTAUTH_SECRET` and `CALENDSO_ENCRYPTION_KEY` from `Cal_Common`. `scripts_dir` is resolved to `abspath("${module.cal_app.path}/scripts")` at apply time.

**GKE-specific note:** Unlike `Cal_CloudRun`, there is no separate `cal.tf` file in `Cal_GKE`. The `Cal_Common` module call and local wiring are defined directly in `main.tf` of `Cal_GKE`.

**PostgreSQL note:** Cal.diy requires **PostgreSQL 15**. `database_type = "POSTGRES_15"` is the default and should not be changed.

## 2. IAM & Access Control

`Cal_GKE` delegates all IAM provisioning to `App_GKE`. Workload Identity is used to bind the Kubernetes pod service account to a GCP service account. The Cloud Build SA, IAP service agent, and password rotation role sets are identical to those in [App_GKE §2](../App_GKE/App_GKE.md#2-iam--access-control).

**Auto-generated application secrets:** `Cal_Common` generates `NEXTAUTH_SECRET` and `CALENDSO_ENCRYPTION_KEY` at apply time using `random_password` (32 characters, no special characters). Both are written to Secret Manager and injected into the GKE pod as secret environment variables.

**Secret naming pattern:** `<application_name>-<deployment_id>-nextauth-secret` and `<application_name>-<deployment_id>-encryption-key`.

**Database initialisation identity:** The `db-init` Kubernetes Job runs under the pod SA bound via Workload Identity. It connects to Cloud SQL PostgreSQL via the Auth Proxy sidecar (since `enable_cloudsql_volume = true` by default), using the PostgreSQL socket path, `DB_USER`, and `DB_PASSWORD` (from Secret Manager).

For the complete role tables and Workload Identity binding details, see [App_GKE §2](../App_GKE/App_GKE.md#2-iam--access-control).

## 7. Platform-Managed Behaviours

| Behaviour | Implementation | Detail |
|---|---|---|
| **PostgreSQL 15 required** | `database_type = "POSTGRES_15"` default | Cal.diy uses Prisma ORM targeting PostgreSQL exclusively. |
| **NEXTAUTH_SECRET auto-generated** | `random_password` in `Cal_Common` | 32-character alphanumeric secret stored in Secret Manager. |
| **CALENDSO_ENCRYPTION_KEY auto-generated** | `random_password` in `Cal_Common` | 32-character alphanumeric key stored in Secret Manager. |
| **NEXT_PUBLIC_WEBAPP_URL uses GKE sentinel** | `$(GKE_SERVICE_URL)` in `main.tf` locals | App_GKE resolves this to the actual external IP at apply time. Override via `environment_variables` for custom domains. |
| **NEXTAUTH_URL mirrors NEXT_PUBLIC_WEBAPP_URL** | Hardcoded in `main.tf` locals | Both set to `$(GKE_SERVICE_URL)` by default. |
| **NODE_ENV = "production"** | Hardcoded in `main.tf` locals | Cannot be overridden via `environment_variables`. |
| **Unix socket by default** | `enable_cloudsql_volume = true` default | Cal.diy connects to Cloud SQL via Auth Proxy sidecar. Enforced by validation when `database_type != "NONE"`. |
| **Default db-init job** | Supplied by `Cal_Common` when `initialization_jobs = []` | PostgreSQL database and user created automatically. |
| **Scripts directory** | `scripts_dir = abspath("${module.cal_app.path}/scripts")` | Scripts sourced from `Cal_Common/scripts`. |
| **30-second secret wait** | `time_sleep` in `Cal_Common` | Prevents secret-read failures during pod startup. |
| **Network discovery** | `module_network_discovery` in `main.tf` | Auto-discovers the `Services_GCP` VPC region and subnet. |
