# Mautic on GKE Autopilot

This document provides a comprehensive reference for the `modules/Mautic_GKE` Terraform module. It covers architecture, IAM, configuration variables, Mautic-specific behaviours, and operational patterns for deploying Mautic on Google Kubernetes Engine (GKE) Autopilot.

---

## 1. Module Overview

Mautic is an open-source marketing automation platform. `Mautic_GKE` is a **wrapper module** built on top of `App_GKE`. It uses `App_GKE` for all GCP and Kubernetes infrastructure provisioning and injects Mautic-specific application configuration, database initialisation, and storage configuration via `Mautic_Common`.

**Key Capabilities:**
*   **Compute**: GKE Autopilot, PHP/Apache container, 2 vCPU / 4 Gi by default. HPA-managed horizontal scaling with `min_instance_count = 1`.
*   **Data Persistence**: Cloud SQL **MySQL 8.0** (not PostgreSQL). NFS (GCE VM or Filestore) for shared media files. GCS `mautic-media` bucket auto-provisioned by `Mautic_Common`.
*   **Security**: Inherits Cloud Armor WAF, IAP (with OAuth credentials), Binary Authorization, and VPC Service Controls from `App_GKE`. `MAUTIC_ADMIN_PASSWORD` is auto-generated by `Mautic_Common` and stored in Secret Manager.
*   **Caching**: Redis **enabled by default** (`enable_redis = true`). When `redis_host` is empty and `enable_nfs = true`, the NFS server IP is used as the Redis host.
*   **Session Affinity**: `session_affinity = "ClientIP"` is the default — Mautic's PHP session handling benefits from sticky routing.
*   **Validation Guards**: `validation.tf` enforces cross-variable consistency at plan time to prevent misconfigured deployments.

**Project & Application Identity**

| Variable | Group | Type | Default | Description |
|---|---|---|---|---|
| `project_id` | 1 | `string` | — | GCP project ID. **Required.** |
| `tenant_deployment_id` | 2 | `string` | `'demo'` | Short suffix appended to all resource names. |
| `support_users` | 2 | `list(string)` | `[]` | Email recipients for monitoring alerts. |
| `resource_labels` | 2 | `map(string)` | `{}` | Labels applied to all provisioned resources. |
| `application_name` | 3 | `string` | `'mautic'` | Base resource name. Do not change after initial deployment. |
| `application_display_name` | 3 | `string` | `'Mautic'` | Human-readable name shown in the GCP Console. |

**Wrapper architecture:** `Mautic_GKE` calls `Mautic_Common` to build an `application_config` object containing Mautic-specific environment variables, probe configuration, and the `db-init` job definition. `module_storage_buckets` carries the `mautic-media` bucket provisioned by `Mautic_Common`. `scripts_dir` is resolved to the `Mautic_Common/scripts` directory at apply time.

**MySQL note:** Mautic requires **MySQL 8.0**. `database_type = "MYSQL_8_0"` is fixed by `Mautic_Common`. PostgreSQL is not supported.

## 2. IAM & Access Control

`Mautic_GKE` delegates all IAM provisioning to `App_GKE`. The GKE workload SA, Cloud Build SA, Workload Identity binding, and password rotation role sets are identical to those in [App_GKE](../App_GKE/App_GKE.md).

**Auto-generated secrets:** `Mautic_Common` generates `MAUTIC_ADMIN_PASSWORD` (24 characters, no special characters), stored in Secret Manager and injected into the pod via Kubernetes `valueFrom`. The `DB_PASSWORD` secret is provisioned automatically by `App_GKE`.

**Database initialisation identity:** The `db-init` Kubernetes Job runs under the workload service account with Workload Identity. It connects to Cloud SQL MySQL via the Auth Proxy Unix socket (`enable_cloudsql_volume = true` by default).

For the complete role tables and IAP, password rotation, and public access details, see [App_GKE](../App_GKE/App_GKE.md).

### E. Initialization & Bootstrap

A `db-init` Kubernetes Job is automatically provisioned by `Mautic_Common` when `initialization_jobs` is left empty. It uses `mysql:8.0-debian` and executes `db-init.sh`.

`DOCKER_MAUTIC_RUN_MIGRATIONS=true` causes the Mautic entrypoint to run database migrations on each pod start. This is safe (idempotent) and ensures schema updates are applied on version upgrades. The startup probe uses a 90-second initial delay and 20 failure threshold to accommodate this.

| Variable | Group | Default | Description |
|---|---|---|---|
| `initialization_jobs` | 11 | `[]` | Kubernetes Jobs. Leave empty for `Mautic_Common` to supply the default `db-init` job. Each job must define `command`, `args`, or `script_path`. Non-empty list replaces the default entirely. |
| `cron_jobs` | 11 | `[]` | Kubernetes CronJobs for scheduled tasks. Critical for campaign processing. |
| `additional_services` | 11 | `[]` | Sidecar or helper GKE services. |

## 7. Cron Jobs for Campaign Processing

Mautic requires scheduled commands for full functionality. Define these via `cron_jobs`:

| Command | Purpose | Recommended Schedule |
|---|---|---|
| `mautic:campaigns:trigger` | Trigger scheduled campaign events | Every 15 minutes |
| `mautic:campaigns:messages` | Send queued campaign messages | Every 15 minutes |
| `mautic:queue:process` | Process email send queue | Every 5 minutes |
| `mautic:contacts:deduplicate` | Deduplicate contacts | Daily |
| `mautic:maintenance:cleanup` | Clean up old data | Weekly |

Example `cron_jobs` configuration:

```hcl
cron_jobs = [
  {
    name     = "campaign-trigger"
    schedule = "*/15 * * * *"
    command  = ["php", "/var/www/html/bin/console", "mautic:campaigns:trigger", "--env=prod"]
    cpu_limit    = "500m"
    memory_limit = "256Mi"
  },
  {
    name     = "queue-process"
    schedule = "*/5 * * * *"
    command  = ["php", "/var/www/html/bin/console", "mautic:queue:process", "--env=prod"]
    cpu_limit    = "500m"
    memory_limit = "256Mi"
  }
]
```

## 8. Validation Guards

`Mautic_GKE/validation.tf` enforces the following cross-variable rules at plan time:

| Rule | Error Condition |
|---|---|
| HPA scaling bounds | `min_instance_count > max_instance_count` |
| Redis host requirement | `enable_redis = true` and `redis_host = ""` and `enable_nfs = false` |
| IAP OAuth credentials | `enable_iap = true` and either OAuth credential is empty |
| CloudSQL volume consistency | `enable_cloudsql_volume = true` and `database_type = "NONE"` |
